# 📋 프로젝트 작업 정리 — YOLOv8n 버튼 검출 (데이터 → 학습)

> 작성일: 2026-06-11
> 목표: 라즈베리파이5 + Hailo-8 가속기용 커스텀 YOLOv8n(버튼 5종 탐지) 모델 제작
> ※ **.hef 변환 단계의 상세 기록은 별도 문서 [`YOLOv8n_Hailo8_변환_작업기록.md`](./YOLOv8n_Hailo8_변환_작업기록.md) 참고.** 이 문서는 그 *앞 단계*(데이터 수집·라벨링·학습)와 전체 흐름을 다룬다.

---

## 프로젝트 개요
- **목표**: 라즈베리파이5 + Hailo-8용 버튼 검출 모델 → 최종 `.hef`
- **모델**: COCO 기반 `yolov8n.pt` → 버튼 5클래스로 전이학습 (출력 head 80→5 교체, backbone 재사용)
- **클래스 5종**: `0=B1(노랑) 1=B2(흰색) 2=B3(핑크) 3=B4(블랙) 4=EMO(빨강 비상정지)`
- **촬영 구성**: 검정 배경 아케이드 버튼 2×2 + 중앙 비상버튼, 손/종이로 가리며 촬영(occlusion 포함)

## 1단계 — 환경 구축 & 프레임 추출 ✅
- sudo가 막혀 있어 **정적 ffmpeg 바이너리**를 `~/.local/bin/`에 설치 (무권한 우회)
- 동영상 2개(1920×1080, 59/60fps, 32.5초 + 55.4초)에서 **fps=2로 176장 추출**
- 위치: `~/projects/frames/` (v1: 65장, v2: 111장)

## 2단계 — 라벨링 전략 수립 ✅
- **과거 기록 분석** (노션 페이지 + 6주차 PDF):
  - SAM3(폴리곤) 회피 → **박스 라벨링**으로 결정 (지난 클린룸 프로젝트 실패 경험)
  - 6주차 PDF의 .hef 변환 실패 3대 원인(입력형식/NMS/클래스수) 파악
- **라벨링 규칙 확정**:
  - 손/종이에 가린 버튼도 라벨 (박스는 버튼 전체 둘레 = amodal)
  - 25% 이상 보이거나 위치로 식별 가능하면 라벨, 거의 안 보이면 생략
  - 색상(hue)·채도·흑백 증강 금지 (색으로 클래스 구분하므로)
- 팀 협업 라벨링(Label With My Team) 흐름 + Null/생략/삭제 기준 안내

## 3단계 — 데이터셋 검증 ✅
Roboflow export(`데이터셋/Object Detection.v1i.yolov8.zip`) 받아 직접 검증:
- 라벨 1589개 **전부 박스 형식(5토큰), 폴리곤 0개** → SAM3 문제 없음 확인
- 클래스 균형 양호 (B1 320 / B2 323 / B3 308 / **B4 342(최다)** / EMO 296)
- train 369 / valid 35 / test 18 (증강 3x), 배경 Null 3장
- `data.yaml`: `nc:5`, `names:[B1,B2,B3,B4,EMO]` 정상

## 4단계 — 학습 & 결과 분석 ✅ (학습은 Colab T4)
- Ultralytics 8.4.63, 100 epoch 설정 → 58 epoch에서 조기종료(best=epoch 38)
- 색 증강 억제 적용: `hsv_h=0.0, hsv_s=0.2, hsv_v=0.3`
- **Test 전체 mAP50 0.993 / mAP50-95 0.835**
- 클래스별 test mAP50: B1 0.991 / B2 0.995 / B3 0.995 / **B4 0.987(최약, 50-95는 0.758)** / EMO 0.995
- **B3↔EMO 혼동 없음** (색 증강 억제 효과)
- ⚠️ 주의: train/val/test가 같은 2영상 랜덤분할이라 프레임이 닮은꼴 → **지표가 낙관적**. 다른 환경 배포 시 새 영상 검증 필요.
- 산출물: `데이터셋/runs/button_v1/weights/best.pt` (6.3MB)

## 5단계 — Hailo 변환 준비 ✅
- 타겟 확정: **Hailo-8 (26 TOPS)** → `--hw-arch hailo8`
- 이 노트북은 **Ubuntu 26.04 + Python 3.14**라 DFC 설치 부적합 → 변환은 별도 PC에서 수행
- **캘리브레이션 123장** 준비 (train/images 중복 제거, v1=51/v2=72) → `데이터셋/extracted/사진/`
- best.onnx export (Colab, opset 11, 640) → 별도 PC로 이동

## 6단계 — .hef 변환 ✅ (별도 PC에서 수행)
> **상세 과정은 [`YOLOv8n_Hailo8_변환_작업기록.md`](./YOLOv8n_Hailo8_변환_작업기록.md) 참고.**

요약만:
- 버전 짝: **DFC 3.33.1 ↔ hailo_model_zoo v2.18** (DFC 5.x는 Hailo-10/15용이라 금지)
- 명령: `hailomz compile yolov8n --classes 5 --hw-arch hailo8 --end-node-names <헤드 conv 6지점>`
- **🎉 결과: `yolov8n.hef` (4.4MB) 생성 성공**
- 과거 3대 실패원인 전부 정상 검증: **UINT8 입력 / NMS 내장 / 5클래스**
- ⚠️ GPU(RTX 5060 Blackwell)가 DFC 3.33.1과 호환 안 돼 CPU 양자화(level 0) → 정확도 최상은 아닐 수 있음

- **완료**: 데이터 수집 → 프레임 추출 → 라벨링 → 학습 → **`.hef` 변환까지 전부 성공**

  - **추론 코드 3원칙** (과거 실패 재발 방지):
    1. 입력 = **uint8 640×640 RGB** 그대로 (float 정규화 ❌)
    2. 출력 = **HailoRT NMS 결과** 파싱 (raw 텐서 디코딩 ❌)
    3. 클래스 순서 = `0=B1(노랑) 1=B2(흰색) 2=B3(핑크) 3=B4(블랙) 4=EMO(빨강)`
    4. Pi의 HailoRT는 **4.x 유지** (DFC 3.33.1 hef와 호환)

## 핵심 교훈 (다음 작업에서도 유효)
1. **SAM3 금지** — 폴리곤 출력이라 YOLO 박스 학습 부적합. Object Detection 박스 라벨링만.
2. **색 증강 금지** — 클래스가 색으로 구분되므로 hue/saturation/grayscale 변형은 라벨 파괴.
3. **.hef ↔ 추론코드 3대 정합성** — 입력형식(UINT8)/NMS 위치/클래스 수가 반드시 일치해야 탐지됨.
4. **버전 매칭** — DFC ↔ Model Zoo ↔ Pi HailoRT 버전이 호환되어야 함. 최신이 항상 옳은 게 아님(Hailo-8은 DFC 3.x 라인).
