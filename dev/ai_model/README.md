# ai_model — 트랙 B (버튼 동적검출 모델)

> 버튼을 1인칭 글래스 영상에서 **동적 검출**하는 YOLO 모델 트랙.
> Step1 PoC([`../poc`](../poc))는 색 ROI로 대체했고, 여기서 ROI를 YOLO 검출로 교체한다.
> 설계 정본: [`docs/통합수행설계문서_전체_섹션1-15.md`](../../docs/통합수행설계문서_전체_섹션1-15.md) §7·§10.

## 모델 단계 (네이밍 규약 정본 §7.1)
| 모델 | 정체 | 상태 |
|---|---|---|
| `person_v1.pt` | 초기 검증(person 단일, mAP50≈0.96) | ✅ 완료 |

| `console_v2` | 최종 시연 모델(정확도 보강 재학습) | ⬜ 추후 |

## console_v1 — 확정 사실 (2026-06-11 빌드)
**5클래스 (버튼만 — 디스플레이·타워램프 미포함):**

| idx | 클래스 | 색 | 단계 라벨(정본 §6.1) |
|---|---|---|---|
| 0 | `B1` | 노랑 | 클린·가스차단 |
| 1 | `B2` | 흰 | 펌프/퍼지 |
| 2 | `B3` | 핑크 | 전극 냉각 |
| 3 | `B4` | 검정 | 챔버 벤트 |
| 4 | `EMO` | 빨강 | 비상정지 |

**입출력 규격 (Pi 추론 시 반드시 지킬 것 — 과거 미탐지 원인):**
- 입력: **uint8 640×640 RGB** (float32 정규화 ❌ — 정규화는 칩 내부에서 수행)
- 출력: **HailoRT NMS 결과** `[클래스5, 5, 100]` 파싱 (raw 텐서 디코딩 ❌)
- **HailoRT 4.x 필수** (DFC 3.33.1로 만든 hef와 호환 / 5.x로 올리면 호환 깨짐)

**⚠️ 정확도 한정:** GPU 없이 캘리브 64장 → 양자화 **optimization level 0**. 동작엔 문제없으나 정밀도 낮을 수 있음, 특히 **B4(검정) 약함**. 부족 시 → GPU + 캘리브 1024+장 재최적화 = **`console_v2`**.

**산출물 위치:** `console_v1.hef`(4.4MB, 빌드 출력 `yolov8n.hef`를 개명)는 현재 **`D:\Hailo_DFC\`** 에 있음(빌드 환경). **아직 repo·파이에 미반영.** 전체 변환 과정·검증은 [`console_v1_Hailo변환기록.md`](console_v1_Hailo변환기록.md).

## 통합 시 주의 (Pi)
- Rpi5 `detector`를 위 입출력 규격(uint8 640 / NMS 파싱)으로 배선.
- ⚠️ **`Rpi5/Demo/recipe.json`의 단계 라벨이 구 기동 시퀀스**(웨이퍼 로딩·진공 배기·공정가스·증착)라 정본 §6.1 **PM 시퀀스**(클린·가스차단·펌프퍼지·전극냉각·챔버벤트)와 불일치 → 통합 전 동기화 필요. (버튼 클래스명 B1~B4·EMO는 양쪽 일치)
- 입력 해상도: console_v1은 **640×640**. 정본 §7.1 "QVGA 320 기본"과 어긋남 → 해상도 정책 확정 필요(별도).

## 향후 단계
1. **(다음)** `console_v1.hef` 파이 통합·실추론 — B1~B4+EMO 검출·FPS 실측 → 정본 갱신.
2. 색 ROI → YOLO ROI 교체, 가림 대응(camera_thread IoU 트래커 `YOLO_MAX_MISS`).
3. 정확도 보강 재학습 → **`console_v2`**(최종 시연).
- 디스플레이·타워램프 검출은 console_v1 미포함 → 필요 시 console_v2/별도 트랙.

## 기술 핵심
SOP = JSON 분리 / 가림 학습 + 트랙 지속 / 재현율 > 정밀도.
