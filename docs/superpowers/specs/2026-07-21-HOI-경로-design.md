# HOI(손 검출) 경로 — 조사·실증 결과와 권고 (2026-07-21)

> 사용자가 test 라벨링을 진행하는 동안의 **병행 작업**. 결정 부담을 주지 않기 위해 조사·측정·권고까지 수행하고, **채택 결정은 사용자 몫으로 남긴다.**
> ⚠️ 이 문서는 논의·미결·권고만 담는다. 확정 수치의 정본은 통합문서 §10 — **아래 측정치는 실시간(카메라) 검증 전이라 아직 §10에 올리지 않았다.**

## 1. 문제 — 왜 막혀 있었나

CLAUDE.md가 "🔴 HOI 보류 — **그런데 이것이 실테스트의 선행조건이다**"로 남긴 미해결 과제. 두 가지를 동시에 막고 있었다:

- **§4 NFR-1 FPS 판정 불가** — §10.6이 요구하는 건 "console_v2 + MediaPipe 포함 FPS 재측정"인데, MediaPipe가 안 돌아 **측정 자체가 불가**했다.
- **사전 차단 불가** — §7.2 HOI 융합이 비전 순서위반 감지의 한 축인데 미동작. 지금 시연하면 **"버튼이 보인다"까지만 되고 손이 어디로 가는지를 몰라 사전 차단이 안 된다.**

> ⚠️ **역할 구분(2026-07-22 정정)** — 비전이 "눌렀다"를 판정하는 것이 아니다. `fsm.py`는 입력이 둘로 나뉜다:
> - **`press_button()`** = **물리 버튼(GPIO)** 이 담당. 주석에도 *"물리 버튼이 실제로 눌렸을 때 호출"*
> - **`update_vision(roi)`** = **비전(HOI)** 이 담당. 손이 머무는 ROI를 넘기면 FSM이 체류 타이머로 **스침 vs 위반**을 가른다. 주석: *"손이 정답 ROI에 있어도 '눌림' 전까지는 진전시키지 않는다(press_button이 확정)"*
>
> 프로젝트 목적이 **"버튼 누르기 직전에 사전 감지·차단"**(CLAUDE.md)이므로, 비전이 답할 질문은 **"눌렀나?"가 아니라 "지금 잘못된 버튼으로 손이 가고 있나?"** 다. 눌림 자체는 이미 GPIO로 감지된다(트랙 A 완료, §12).

원인: 파이가 **Python 3.13 단일**(`hailo_platform`이 3.13용 시스템 패키지)인데 **mediapipe는 3.13/aarch64 휠이 없다.** 2026-07 시점에도 여전히 미제공이며, 공식 빌드 인프라도 3.9~3.12만 지원한다. `apt`에 python3.11/3.12도 없어 별도 venv 방식은 **소스 빌드가 선행**돼야 한다.

## 2. 해법 — MediaPipe를 안 쓰고 MediaPipe 모델을 쓴다

`AlbertaBeef/blaze_app_python`의 `blaze_hailo` 타깃이 **BlazePalm(손바닥 검출) + BlazeHandLandmark(21점)** 을 Hailo에서 돌린다. MediaPipe의 손 파이프라인을 그대로 이식한 것이다.

**핵심**: 의존성이 `hailo_platform` + `numpy` + `cv2` **뿐**이다(소스 확인). **mediapipe pip 패키지가 아예 불필요**하므로 Python 3.13 문제가 **원천 소멸**한다.

- 모델 = `blaze_hailo8_models.zip`(23.7MB, GitHub 릴리스 `hailo8_version_2`) — **Hailo-8 전용 빌드**(8L 아님)
- `hailortcli parse-hef` 검증: 둘 다 `HAILO8` 컴파일, HailoRT **4.23.0**에서 정상 파싱
- `hand_landmark_lite` 출력 `fc1: NC(63)` = **21점 × 3좌표** — 설계(§7.2)가 요구하는 랜드마크 수와 일치

## 3. 실증 결과 (파이 실측)

도구 = `Rpi5/Demo/test/hoi_probe.py`(신규, 독립 실행 — **런타임 코드 미변경**). 입력은 기존 raw 프레임 재사용(새 촬영 없음).

### ✅ 손 검출이 실제로 된다
`154947`(가림) 세션 `f00203`: **손 1개 · 랜드마크 (21,3) · score 0.983 · 46ms**

⭐ **클린룸 흰 장갑을 낀 손인데도 검출됐다.** 실배포 조건이 장갑이므로 중요한 확인이다.

### ✅ 좌표계가 console_v2와 일치한다
시각화(`--viz`)로 확인 — 손 랜드마크와 console_v2 버튼 박스가 **같은 640×480 좌표계**에 정합. 별도 변환 없이 접촉 판정이 가능하다. 접촉 기준점 = 랜드마크 **인덱스 8(검지 끝)**.

### ✅ 검출이 진짜 신호다 (오검출 아님)

| 세션 | 성격 | 손 검출률 | score |
|---|---|---|---|
| `154947` 가림 | 손이 자주 등장 | **38.0%** (114/300) | median 0.854 |
| `154730` 각도 | 손 거의 없음(대조군) | **0.3%** (1/300) | 0.027 |

대조군의 유일한 검출도 score 0.027로 임계값(0.5) 미달 → **실질 오검출 0**.

**가림 정도와의 상관**(같은 세션 내부, 임계 0.5 적용):

| console_v2 버튼 검출 수 | 프레임 | 손 검출률 |
|---|---|---|
| 1개 (가림 심함) | 3 | 66.7% |
| 3개 | 59 | 50.8% |
| 4개 | 115 | 36.5% |
| 5개 (가림 없음) | 101 | **13.9%** |

버튼이 가려질수록 손 검출률이 오른다 — **손 검출기가 진짜 손을 찾고 있다는 증거**다.

### ✅ 3모델 동시 구동 + NFR-1 통과

스케줄러(`HailoSchedulingAlgorithm.ROUND_ROBIN`)로 한 VDevice에 3모델을 올려 순수 추론 시간 측정:

| 모델 | 추론 | 상당 FPS |
|---|---|---|
| console_v2 | 9.5ms | 104.8 |
| palm_detection_lite | 5.7ms | 175.3 |
| hand_landmark_lite | 2.9ms | 350.0 |
| **3모델 직렬 합** | **18.1ms** | **55.2** |

**NFR-1 목표 15fps 대비 3.7배 여유.** 손 검출을 얹어도 **모델은 병목이 아니다.**

⚠️ **이 수치의 한계**: NPU 순수 추론 시간이며 CPU 전·후처리(리사이즈·NMS 디코드·ROI 어파인)와 **ESP32 TCP 전송**이 빠져 있다. 어제 실측(§10.18)에서 실시간 FPS는 **9~12.6**이었고 추론은 11ms에 불과했다 — 즉 **실제 병목은 카메라 전송(§10.11)이지 모델이 아니다.** 손 검출 추가분(+8.6ms)은 프레임 간격(약 80ms)보다 훨씬 작아 **실시간 FPS를 크게 떨어뜨리지 않을 것으로 예상**되나, **카메라 연결 후 실측으로 확인해야 한다.**

### 🟡 손끝 정확도 — 기하학적으로는 정확, ROI 신호의 안정성은 재확인 필요 (2026-07-21 추가 검증)

> 이 절은 문서 최초 작성 후에 수행한 검증이다. 아래 §6-4가 "미검증"으로 남겨두었던 항목의 답이다.

**방법**: 정답 라벨이 없으므로 ①손끝(랜드마크 8번)과 console_v2 버튼 박스의 거리를 계산하고 ②시각화로 눈으로 확인하고 ③연속 프레임에서 안정성을 추적했다. `154947`(가림) 세션 300장 중 **손 score ≥ 0.7 이면서 버튼도 검출된 프레임 77개**가 대상.

#### ✅ 랜드마크는 실제 손가락을 정확히 따라간다

손끝이 버튼 박스 **안**에 들어간 프레임이 5개. 그중 시각화로 확인한 결과:

| 프레임 | 최근접 버튼 | 거리 | 손 자세 | 판정 |
|---|---|---|---|---|
| **132** | **B1** | **12px** | **검지를 뻗어 B1을 가리킴** | ⭐ 마커가 정확히 B1 박스 위 |
| 76 | B2 | 9px | 손을 편 채 지나감 | 뻗은 손가락 방향의 버튼 |
| 68 | EMO | 13px | 손을 편 채 지나감 | 〃 |
| 300 | B3 | 13px | 손을 편 채 움직임 | 〃 |

**프레임 132가 결정적이다** — 검지가 명확히 뻗어 B1을 향하고, 손끝 마커가 그 위에 있다. **랜드마크 자체는 신뢰할 수 있다.**

#### 🟡 그러나 ROI 신호가 프레임마다 흔들린다

두 가지 문제를 확인했다.

**① 손을 편 채 지나가도 손끝이 버튼에 들어간다.** 위 표의 76·68·300은 손가락을 편 채 이동하는 자세인데도 손끝이 박스 안에 들어왔다. 누르는 게 아닌데 접촉으로 읽힌다.

**② 손끝이 프레임마다 크게 튄다.** 연속 프레임(125~145) 추적 결과:

| 프레임 | 최근접 버튼 | 이동량 |
|---|---|---|
| 132 | **B1** (12px) | 66px |
| 133 | **EMO** | 53px |
| 134 | **B2** | 44px |
| 135~136 | B2 | 31 → 21px |
| 137~141 | **EMO** | — |
| 142 | **B2** | 23px |

**5프레임 만에 B1 → EMO → B2로 바뀐다.** 게다가 126·127·129·130·143·145는 **손 검출 자체가 끊긴다.**

#### 의미 — 이미 알려진 현상이며 대책도 설계에 있다. 다만 **새 검출 수단에서 재확인**이 필요하다

⚠️ **정정(2026-07-22)** — 처음엔 "체류 임계를 정할 데이터가 없다"고 적었으나 **틀렸다.** §9.4 임계값 정본 표에 **이미 실측값이 있다**:

| §9.4 정본 | 값 | 근거 |
|---|---|---|
| 타이머 체류시간 임계 | **dwell 0.5초 + 갭메우기 0.3초** | ✅ PoC 실측 — **스침 오탐 0%**(§10.3) |

특히 **갭메우기 0.3초**가 핵심이다. §9.4 설명: *"누름 순간 손이 버튼을 가려 인식이 끊기는 채터를 메우는 필수 요소"*.

**위에서 관찰한 "검출이 5프레임 만에 끊긴다"가 바로 그 채터다.** 새로 발견한 결함이 아니라 **PoC 때 이미 확인되고 대책까지 마련된 현상**이다. 갭메우기를 적용하면 끊긴 구간이 메워져 체류가 이어질 수 있다.

**그래서 남은 질문은 "임계를 처음부터 정하는 것"이 아니라 "PoC 값이 새 수단에도 유효한가"다.** 0.5초 값은 **색-ROI PoC** 기준이고, 지금 검출 수단은 **console_v2 + HOI 손 랜드마크**로 바뀌었다.

⚠️ **위 측정 데이터는 불리한 조건이다.** `154947`은 **가림을 만들려고 손을 크게 휘저은** 촬영이지 버튼을 조작하는 장면이 아니다. 실제로 누르는 동작에서는 손이 목표 버튼 앞에서 **멈추므로** 양상이 다를 것이다.

→ **▶ "버튼 누르는 동작" 촬영으로 확인할 것**(30초):
①누르는 동안 손끝이 한 버튼에 머무는 시간 → 0.5초로 충분한가
②검출이 끊기는 구간의 길이 → 갭메우기 0.3초로 메워지는가
③접근 중 다른 버튼을 스치는 빈도 → 오탐 위험

§9.4에 **재측정 계획이 이미 문서화**돼 있다 — *"물리 버튼 전장 구성이 완료되면 … 실제 작업처럼 시뮬레이션 측정해 console_v2·실물 기준 임계값을 재설정한다"*. 이 촬영이 그 계획의 실행이다.

#### ⭐ 원인 가설 — 흔들림은 모델 탓이 아니라 **빠진 기능** 탓이다 (2026-07-22 추가)

`blaze_app_python`의 구현을 확인한 결과 **프레임 간 트래킹과 시간축 스무딩이 없다.** `blaze_detect_live.py`가 팜 검출(`predict_on_image`)을 **매 프레임 무조건 실행**하고, `prev_` 상태나 필터가 전혀 없다 — **매 프레임이 서로 독립**이다.

반면 **MediaPipe 원본은 그렇게 동작하지 않는다**(공식 문서 확인):

> 팜 검출기는 매 프레임 실행되지 않는다. 연속 프레임에서는 **이전 프레임의 랜드마크 예측으로부터** 바운딩 박스를 유도하고, 검출기는 **최초 프레임이거나 트래킹 신뢰도를 잃었을 때만** 재실행된다.
> — [Google Research](https://research.google/blog/on-device-real-time-hand-tracking-with-mediapipe/) · [MediaPipe Hand Landmarker 가이드](https://ai.google.dev/edge/mediapipe/solutions/vision/hand_landmarker)

즉 **우리가 가져온 것은 모델(①②)이고, MediaPipe가 하던 트래킹·스무딩(⑥⑦)은 빠져 있다.**

| MediaPipe 프레임워크 구성 | 우리 경로 |
|---|---|
| ① 팜 검출 모델 / ② 핸드 랜드마크 모델 | ✅ **동일**(구글 원본을 `.hef`로 변환) |
| ③ 앵커 디코딩 / ④ NMS / ⑤ ROI 어파인 | ✅ `blaze_common`이 재구현 |
| ⑥ **프레임 간 트래킹** | ❌ **없음** |
| ⑦ **시간축 스무딩** | ❌ **없음** |

**이것이 위 흔들림을 설명한다**:
- *5프레임 만에 B1→EMO→B2* → 매 프레임 독립 추정이라 손이 조금만 움직여도 결과가 튄다
- *126·127·129·130 검출 끊김* → 트래킹이 없어 **한 번 놓치면 그대로 공백**이 된다(MediaPipe는 이전 랜드마크로 이어간다)

→ **모델의 한계가 아니라 채워 넣을 수 있는 성격의 문제다.** 그리고 채울 수단이 이미 프로젝트 안에 둘 있다:
1. **§9.4의 갭메우기 0.3초** — *"인식이 끊기는 채터를 메우는 필수 요소"*. 정확히 이 문제에 대한 대책이며 PoC에서 검증됐다
2. **`camera_thread.py:_update_tracks()`** — 버튼 검출에 이미 쓰는 IoU 간이 트래킹(`YOLO_MAX_MISS` 초과 시 제거). 같은 구조를 손에도 적용할 수 있다

**단 이것은 가설이다.** 갭메우기를 적용하면 체류가 실제로 이어지는지 **측정으로 확인해야 한다** → 아래 §7 도구.

#### 🔴 부수 발견 — 코드와 §9.4 정본이 어긋나 있다

체류 임계 관련 숫자가 **세 개**이고 의미가 각각 다르다:

| 값 | 의미 | 위치 | 상태 |
|---|---|---|---|
| **0.5초 + 갭메우기 0.3초** | 비전 dwell 임계 | **§9.4 정본** | ✅ PoC 실측 |
| **1.0초** | 〃 | **`config.py:57` `FSM_DWELL_THRESHOLD_SEC`** | 🔴 **폐기된 초안값** |
| 1.5초 | **사람이 버튼까지 이동하는 소요시간** | §9.4 재측정 계획 | 잠정 — **축이 다름**(병기 주의) |

`config.py:56` 주석은 *"§9.4: 0.8~1.0초 범위 → 데모 튜닝 시 단일값 확정"* 이라 적혀 있으나, **§9.4는 그 값을 명시적으로 폐기했다**: *"통합설계 초안의 0.8~1.0초는 이 PoC 실측값으로 대체되었다."* → **코드가 폐기된 초안값을 쓰면서 §9.4를 근거로 인용**하고 있다. 또한 **갭메우기 0.3초는 구현에 없다.**

**조치 결정(2026-07-22): 지금은 기록만 한다.** `FSM_DWELL_THRESHOLD_SEC`은 시연 동작을 바꾸는 값이고, 어차피 위 시뮬레이션 촬영 후 재설정할 값이라 **두 번 건드리지 않는다.** 촬영·측정 후 코드·정본·갭메우기 구현을 한 번에 정리한다.

### ⚠️ 정정 — HOI 융합·ROI 판정은 **이미 구현돼 있다** (2026-07-22)

이 문서 초안이 "HOI 융합 로직 미작성"으로 적었으나 **틀렸다.** 확인 결과 파이프라인이 이미 다 있다:

| 구성 | 위치 | 상태 |
|---|---|---|
| 손끝 → ROI 판정 | `camera_thread.py:103 roi_at_point()` | ✅ 구현 |
| ROI → FSM 전달 | `camera_thread.py:385 roi_signal` → `update_vision()` | ✅ 배선 |
| 단위 테스트 | `test_hoi_sim.py` | ✅ 존재 |
| **손 좌표 공급** | `camera_thread.py:376` MediaPipe `INDEX_FINGER_TIP` | 🔴 **동작 안 함** |

**ROI = 검출 박스 그 자체**이며 이는 의도된 설계다. `roi_at_point` 주석:

> ROI는 고정 좌표가 아니라 **YOLO가 검출한 버튼 박스 그 자체**다(FPV에서 박스가 버튼을 따라다니므로 화면이 움직여도 유효). 박스가 겹치면 더 작은(가까운) 박스를 우선해 오판을 줄인다.

웨어러블 1인칭이라 **고정 ROI(`dev/poc/rois.json` 방식)는 성립하지 않는다** — 고개를 돌리면 화면 속 버튼 위치가 통째로 바뀐다. 검출 박스가 따라다니는 것이 유일하게 맞는 방법이다.

→ **따라서 실제로 비어 있는 것은 "손 좌표 공급" 하나뿐이다.** MediaPipe가 안 돌아 `fingertip`이 항상 `None`이고, 그래서 `roi_at_point`가 호출조차 되지 않는다. **Hailo 손 검출을 붙여 `fingertip`을 채우면 나머지 파이프라인은 그대로 동작한다** — 통합 범위가 당초 예상보다 훨씬 작다.

## 4. 🔴 통합 시 반드시 넘어야 할 제약

**`VDevice()`는 물리 장치당 하나만 만들 수 있다.** 두 번째 생성 시 `HAILO_OUT_OF_PHYSICAL_DEVICES`(error 74)로 실패한다 — 실측 확인.

현재 `detector.py:HailoDetector`는 자체 `VDevice()`를 만들고 `network_group.activate()`를 `ExitStack`에 **영구 보유**한다. 따라서:

- ❌ 손 검출을 **별도 VDevice로 붙이는 방식은 불가능**
- ✅ **하나의 VDevice를 공유**해 3모델을 올리는 방식은 **동작 확인됨**

→ 통합하려면 `detector.py`가 **외부에서 주입받은 VDevice/추론 핸들을 쓰도록** 구조를 바꿔야 한다. 이는 작동 중인 코드의 변경이므로 **승인 사안**이며, 이번 병행 작업에서는 **하지 않았다.**

## 7. 시뮬레이션 측정 도구 — `test/dwell_probe.py` (2026-07-22 구현)

§9.4 임계가 새 검출 수단에서도 유효한지 **촬영본으로 바로 답할 수 있게** 미리 만들어 둔 도구. 촬영이 끝나면 **명령 한 줄**로 결과가 나온다.

```bash
python3 test/dwell_probe.py test/raw/<세션> [--dwell 0.5] [--gap-fill 0.3] [--margin 10]
```

**측정 항목** — §9.4가 요구하는 것 그대로:
1. **ROI 시계열** — 프레임마다 손끝(랜드마크 8번)이 어느 버튼에 있는가
2. **공백 분포** — 검출이 끊긴 구간 길이. **갭메우기 0.3초로 메워지는가**가 핵심 질문
3. **체류 구간** — **갭메우기 전/후로 나눠** dwell 임계를 넘는 구간이 생기는지 비교

**설계 요점**
- **ROI 판정을 런타임과 동일 규칙으로 맞췄다**(2026-07-22) — `roi_of()`가 `camera_thread.roi_at_point()`와 같이 **박스 안 포함 + 겹치면 더 작은 박스 우선**. 측정 도구가 다른 규칙을 쓰면 그 수치가 실제 동작을 대표하지 못한다. **두 구현을 7개 좌표(겹침 케이스 포함)로 대조해 전부 일치 확인.** `--margin`은 실험용이며 기본 0 = 런타임과 완전 동일
- 버튼 박스는 **이미 기록된 `rawdet` 로그에서 읽는다** — console_v2 재추론이 불필요하고, 따라서 **VDevice 충돌이 없다**(손 모델만 올린다)
- 시간은 로그의 **실제 타임스탬프**를 쓴다 — FPS를 가정하지 않으므로 체류시간이 초 단위로 정확하다
- 갭메우기는 `A … None … A` 형태에서 공백이 임계 이하일 때만 채운다. 서로 다른 ROI 사이는 메우지 않는다(없는 체류를 만들어내지 않기 위함)

**검증**(가림 세션 `154947`, 295프레임·30.8초):
- `--margin` 0/30/100 에서 ROI 판정률이 1.7% → 9.5% → 27.1%로 **단조 증가**하고 손 검출률(32.2%)에 수렴 — 판정 로직 정상
- dwell 0.5초 이상 구간 **0개**, 최장 체류 0.08초. 공백은 median **1.19초**라 갭메우기 0.3초로 **한 프레임도 메워지지 않았다**

⚠️ **이 결과를 "임계가 부적절하다"로 읽으면 안 된다.** `154947`은 **가림을 만들려고 손을 크게 휘저은** 촬영이라 애초에 조작 장면이 아니다. 도구가 첫 번째 해석으로 *"이 세션이 조작 장면이 아님"* 을 제시하는 이유다. **이 값은 하한 기준선**이며, 판단은 **버튼 누르는 동작 촬영본**으로 한다.
