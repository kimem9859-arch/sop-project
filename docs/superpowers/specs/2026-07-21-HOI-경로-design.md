# HOI(손 검출) 경로 — 조사·실증 결과와 권고 (2026-07-21)

> 사용자가 test 라벨링을 진행하는 동안의 **병행 작업**. 결정 부담을 주지 않기 위해 조사·측정·권고까지 수행하고, **채택 결정은 사용자 몫으로 남긴다.**
> ⚠️ 이 문서는 논의·미결·권고만 담는다. 확정 수치의 정본은 통합문서 §10 — **아래 측정치는 실시간(카메라) 검증 전이라 아직 §10에 올리지 않았다.**

## 1. 문제 — 왜 막혀 있었나

CLAUDE.md가 "🔴 HOI 보류 — **그런데 이것이 실테스트의 선행조건이다**"로 남긴 미해결 과제. 두 가지를 동시에 막고 있었다:

- **§4 NFR-1 FPS 판정 불가** — §10.6이 요구하는 건 "console_v2 + MediaPipe 포함 FPS 재측정"인데, MediaPipe가 안 돌아 **측정 자체가 불가**했다.
- **손-버튼 접촉 판정 불가** — §7.2 HOI 융합이 비전 순서위반 감지의 한 축인데 미동작. 지금 시연하면 "버튼이 보인다"까지만 되고 **"눌렀다"를 모른다.**

원인: 파이가 **Python 3.13 단일**(`hailo_platform`이 3.13용 시스템 패키지)인데 **mediapipe는 3.13/aarch64 휠이 없다.** 2026-07 시점에도 여전히 미제공이며, 공식 빌드 인프라도 3.9~3.12만 지원한다. `apt`에 python3.11/3.12도 없어 별도 venv 방식은 **소스 빌드가 선행**돼야 한다.

## 2. 해법 — MediaPipe를 안 쓰고 MediaPipe 모델을 쓴다

`AlbertaBeef/blaze_app_python`의 `blaze_hailo` 타깃이 **BlazePalm(손바닥 검출) + BlazeHandLandmark(21점)** 을 Hailo에서 돌린다. MediaPipe의 손 파이프라인을 그대로 이식한 것이다.

**핵심**: 의존성이 `hailo_platform` + `numpy` + `cv2` **뿐**이다(소스 확인). **mediapipe pip 패키지가 아예 불필요**하므로 Python 3.13 문제가 **원천 소멸**한다.

- 모델 = `blaze_hailo8_models.zip`(23.7MB, GitHub 릴리스 `hailo8_version_2`) — **Hailo-8 전용 빌드**(8L 아님)
- `hailortcli parse-hef` 검증: 둘 다 `HAILO8` 컴파일, HailoRT **4.23.0**에서 정상 파싱
- `hand_landmark_lite` 출력 `fc1: NC(63)` = **21점 × 3좌표** — 설계(§7.2)가 요구하는 랜드마크 수와 일치

## 3. 실증 결과 (파이 실측)

도구 = `Rpi5/Demo/test/hoi_probe.py`(신규, 독립 실행 — **런타임 코드 미변경**). 입력은 기존 raw 프레임 재사용(새 촬영 없음).

### ✅ 손 검출이 실제로 된다
`154947`(가림) 세션 `f00203`: **손 1개 · 랜드마크 (21,3) · score 0.983 · 46ms**

⭐ **클린룸 흰 장갑을 낀 손인데도 검출됐다.** 실배포 조건이 장갑이므로 중요한 확인이다.

### ✅ 좌표계가 console_v2와 일치한다
시각화(`--viz`)로 확인 — 손 랜드마크와 console_v2 버튼 박스가 **같은 640×480 좌표계**에 정합. 별도 변환 없이 접촉 판정이 가능하다. 접촉 기준점 = 랜드마크 **인덱스 8(검지 끝)**.

### ✅ 검출이 진짜 신호다 (오검출 아님)

| 세션 | 성격 | 손 검출률 | score |
|---|---|---|---|
| `154947` 가림 | 손이 자주 등장 | **38.0%** (114/300) | median 0.854 |
| `154730` 각도 | 손 거의 없음(대조군) | **0.3%** (1/300) | 0.027 |

대조군의 유일한 검출도 score 0.027로 임계값(0.5) 미달 → **실질 오검출 0**.

**가림 정도와의 상관**(같은 세션 내부, 임계 0.5 적용):

| console_v2 버튼 검출 수 | 프레임 | 손 검출률 |
|---|---|---|
| 1개 (가림 심함) | 3 | 66.7% |
| 3개 | 59 | 50.8% |
| 4개 | 115 | 36.5% |
| 5개 (가림 없음) | 101 | **13.9%** |

버튼이 가려질수록 손 검출률이 오른다 — **손 검출기가 진짜 손을 찾고 있다는 증거**다.

### ✅ 3모델 동시 구동 + NFR-1 통과

스케줄러(`HailoSchedulingAlgorithm.ROUND_ROBIN`)로 한 VDevice에 3모델을 올려 순수 추론 시간 측정:

| 모델 | 추론 | 상당 FPS |
|---|---|---|
| console_v2 | 9.5ms | 104.8 |
| palm_detection_lite | 5.7ms | 175.3 |
| hand_landmark_lite | 2.9ms | 350.0 |
| **3모델 직렬 합** | **18.1ms** | **55.2** |

**NFR-1 목표 15fps 대비 3.7배 여유.** 손 검출을 얹어도 **모델은 병목이 아니다.**

⚠️ **이 수치의 한계**: NPU 순수 추론 시간이며 CPU 전·후처리(리사이즈·NMS 디코드·ROI 어파인)와 **ESP32 TCP 전송**이 빠져 있다. 어제 실측(§10.18)에서 실시간 FPS는 **9~12.6**이었고 추론은 11ms에 불과했다 — 즉 **실제 병목은 카메라 전송(§10.11)이지 모델이 아니다.** 손 검출 추가분(+8.6ms)은 프레임 간격(약 80ms)보다 훨씬 작아 **실시간 FPS를 크게 떨어뜨리지 않을 것으로 예상**되나, **카메라 연결 후 실측으로 확인해야 한다.**

## 4. 🔴 통합 시 반드시 넘어야 할 제약

**`VDevice()`는 물리 장치당 하나만 만들 수 있다.** 두 번째 생성 시 `HAILO_OUT_OF_PHYSICAL_DEVICES`(error 74)로 실패한다 — 실측 확인.

현재 `detector.py:HailoDetector`는 자체 `VDevice()`를 만들고 `network_group.activate()`를 `ExitStack`에 **영구 보유**한다. 따라서:

- ❌ 손 검출을 **별도 VDevice로 붙이는 방식은 불가능**
- ✅ **하나의 VDevice를 공유**해 3모델을 올리는 방식은 **동작 확인됨**

→ 통합하려면 `detector.py`가 **외부에서 주입받은 VDevice/추론 핸들을 쓰도록** 구조를 바꿔야 한다. 이는 작동 중인 코드의 변경이므로 **승인 사안**이며, 이번 병행 작업에서는 **하지 않았다.**

## 5. 권고

**Hailo 팜+핸드 경로를 채택할 것.** 근거:

1. Python 3.13 문제를 **우회가 아니라 소멸**시킨다(mediapipe 의존 자체가 없음)
2. §7.2 설계가 요구하는 **21점 랜드마크를 그대로** 제공 — 설계 변경 불필요
3. 성능이 문제되지 않는다(55fps, 목표의 3.7배)
4. **장갑 낀 손**에서 동작 확인 — 실배포 조건 충족
5. 좌표계가 이미 정합 — 융합 로직이 단순

### 거부한 대안

| 대안 | 거부 사유 |
|---|---|
| `yolov8s_pose_h8.hef`(파이에 기설치) | COCO 17관절은 **손목까지이고 손끝이 없다.** 버튼(약 30px)에 대한 접촉 판정에 부정확. 주 경로가 성공해 불필요해짐 |
| 별도 venv(py3.11)+IPC | apt에 python3.11/3.12 부재 → 소스 빌드 선행. 난이도·유지보수 비용이 크고, 주 경로가 이를 불필요하게 만듦 |
| 가림 자체를 접촉 신호로 사용 | console_v2 검출이 사라지는 것으로 접촉을 추정 — 구현은 0이지만 **손 아닌 물체의 가림과 구분 불가**. 안전 기능의 근거로 부적합 |

## 6. ▶ 다음 결정 지점 (사용자)

1. **채택 여부 결정** — 채택 시 `detector.py`의 VDevice 공유 구조 변경이 필요(설계 승인 후 진행)
2. **카메라 연결 후 실시간 FPS 재측정** — §10.6이 요구하는 "MediaPipe 포함 FPS"의 실제 답. 지금 수치는 모델 상한일 뿐
3. **접촉 판정 임계 설계** — 검지 끝이 버튼 박스에 얼마나 들어와야 "접촉"인가. §9.4의 체류시간(1.0초)과 결합
4. **랜드마크 정확도 평가** — 이번엔 "손을 찾는가"만 확인했다. **손끝이 정확한 버튼을 가리키는가**는 정답 라벨이 없어 미검증. 필요하면 별도 소규모 라벨링
