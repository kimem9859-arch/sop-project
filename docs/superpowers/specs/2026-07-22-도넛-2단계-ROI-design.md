## 1. 배경 — 왜 이것이 필요한가

같은 날 오전 `2026-07-22-손검출률-개선-design.md`가 다음을 확정했다:

- 손 트래킹·ROI 유도는 **효과 0**(폐기)
- 진짜 원인은 **신뢰도 게이트가 정확한 검출을 버리는 것**
- `HAND_MIN_SCORE` 0.5→0.2로 56%→75%
- 🔴 **손 검출 파라미터만으로는 88%가 한계**이고 90% 미달

남은 실패는 **물리적**이었다 — 누르는 순간 손가락이 버튼을 덮어 ①버튼 박스가 사라지고 ②손끝이 박스를 벗어난다. 박스를 넓히면(margin) 감지가 오르지만 **오경보가 3배**가 됐다. 넓힌 구역을 안쪽과 **똑같이** 취급했기 때문이다.

**도넛 구조는 이 트레이드오프의 구조 자체를 바꾼다** — 넓히되 **반응을 단계로 나눈다.** FSM에 `WARNING`·`BLOCK`이 이미 있으므로 새 상태도 필요 없다.

## 2. 측정 — 3세션 실촬영 (2026-07-22)

⚠️ 이전 세션(눌림 16회)은 동작이 빨라 대표성이 의심됐고, **오경보 계산도 오염돼 있었다**(촬영 시 요청한 "오답 버튼 왕복"을 오경보로 세고 있었다). 그래서 3세션을 새로 찍었다.

| 세션 | 눌림 | 용도 |
|---|---|---|
| ① `press-sim-slow` | 26회 | 튜닝 |
| ② `falsealarm-clean` | 23회 | **정상 순서만** 촬영 → 진짜 오경보 측정 |
| ③ `press-sim-slow-holdout` | 29회 | 🔒 최종 판정 (**미개봉**) |

### 2-1. ⭐ 짧았던 선행시간은 사람이 아니라 **우리 검출이 끊긴 것**이었다

| conf | 링폭 | 갭메우기 | 개입 감지(①) | 선행 median |
|---|---|---|---|---|
| 0.5 | 0 | 0 | 13/26 (50%) | 0.24s |
| 0.2 | 25 | 0 | 22/26 (85%) | 0.38s |
| **0.2** | **25** | **0.3** | **24/26 (92%)** | **0.55s** |

**갭메우기 0.3초를 넣는 것만으로 선행 median이 0.38s → 0.55s로 뛴다.** 손은 버튼 위에 계속 있었는데 검출이 잘게 끊겨 "연속 구간"이 쪼개져 있었다. 그 0.3초는 **§9.4 정본값 그대로**다.

🔴 **앞 문서의 "갭메우기 효과 0"은 폐기하지 말고 조건을 붙여 읽어야 한다** — 그것은 **링 없이 "눌리는 그 프레임" 지표로만** 봤을 때의 결과다. 도넛과 결합하면 결정적이다. **두 지표(체류 구간 / 눌림 프레임 / 선행시간)가 서로 다른 것을 재고 있다.**

### 2-2. 링 폭 — 25px가 무릎점

| 링폭 | 개입 감지(①) | 선행 median | ≥0.5s | 오경보(②) |
|---|---|---|---|---|
| 0 | 62% | 0.23s | 4회 | **0개** |
| 10 | 85% | 0.45s | 9회 | 1개 |
| 15 | 88% | 0.50s | 10회 | 1개 |
| **25** | **92%** | **0.55s** | **14회** | **1개** |
| 40 | 92% | 0.56s | 14회 | **2개** |

40px는 감지가 안 늘면서 오경보만 2배가 된다.

### 2-3. 오경보 1건의 정체 — 검출 오류가 아니다

```
B2 위에 1.02초 체류(그중 5프레임은 박스 안쪽) → 2.42초 뒤 B3 누름
score 0.65~0.90
```

B3로 가는 길에 손이 B2 위를 **1초간 천천히 지나갔다.** 시스템 관점에서는 "기대 단계가 아닌 버튼에 1초 체류"라 **경고하는 것이 오히려 정상 동작**이다. 순수한 헛경보가 아니라 **판단이 애매한 사람 동작**이며, 분당 1.1회는 감수 가능하다고 본다.

### 2-4. C1 vs C2 — C1 확정

발화 조건을 두 가지로 시뮬레이션했다.

| 모드 | 발화 조건 | 결과 |
|---|---|---|
| **C1** | 임계 도달 **+ 안쪽에 있을 때만** | 검출 동일, 오경보 **8개** |
| C2 | 임계 도달하면 **링에서도** | 검출 동일, 오경보 **18~19개** |

⚠️ 이 절대값은 오염된 이전 세션 기준이라 쓰지 않는다. 그러나 **C1이 C2보다 2배 이상 낫다는 비교는 유효**하다. **C1으로 확정.**

## 3. 설계

### 3-1. 2단계 구역

```
        ┌─────────────────────┐
        │   1단계 (링, 25px)   │  ← 접근 = 체류 예열 + 경고 표시
        │   ┌─────────────┐   │
        │   │  2단계      │   │  ← 검출 박스 그 자체 = 위험
        │   │  (버튼 박스) │   │     여기서만 차단이 발화한다 (C1)
        │   └─────────────┘   │
        └─────────────────────┘
```

`camera_thread.roi_at_point(fx, fy, tracks)` → **`(버튼, 단계)`** 반환으로 확장.
- 우선순위: **안쪽이 링을 이긴다.** 같은 단계끼리 겹치면 **더 작은 박스 우선**(현행 규칙 유지)
- 링 폭 = `config.HAND_ROI_RING_PX` (기본 **25**)
- ⚠️ `test/dwell_probe.roi_of`도 **같은 규칙**을 유지해야 한다 — 그 파일 주석이 "런타임과 동일해야 한다"고 못 박고 있다

### 3-2. FSM — 예열과 발화의 분리 (C1)

`fsm.py`의 `update_vision(roi, now)` → **`update_vision(roi, now, level=2)`**
- 🔴 **`level=2` 기본값**을 둬서 기존 호출·테스트 15개가 그대로 통과하게 한다(하위 호환).

규칙:
1. **체류 누적** — `roi`가 같은 버튼이면 **단계와 무관하게** 타이머가 쌓인다 (= 예열)
2. **차단 발화** — 누적이 임계를 넘고 **`level == 2`(안쪽)일 때만** `WARNING`/`BLOCK`으로 전이
3. **경고 표시** — `level == 1`은 안돈·로그로만 알린다. **전기 차단 없음**
4. **갭메우기** — `roi`가 `None`이어도 마지막 관측 후 `FSM_GAP_FILL_SEC`(0.3) 이내면 **직전 (버튼, 단계)를 유지**한다. 그 이후엔 리셋

### 3-3. config 변경

| 값 | 현재 | 변경 | 근거 |
|---|---|---|---|
| `HAND_MIN_SCORE` | 0.5 | **0.2** | 정확한 검출이 버려지고 있었다(앞 문서 §6-2). 0.2 아래는 포화 |
| `HAND_ROI_RING_PX` | (없음) | **25** | §2-2 무릎점 |
| `FSM_GAP_FILL_SEC` | (없음) | **0.3** | §9.4 정본. 미구현이었다 |
| `FSM_DWELL_THRESHOLD_SEC` | 1.0 | **0.5** | §9.4 정본. **코드-정본 불일치를 이 작업에서 해소** |

## 4. 검증

**오프라인 (카메라 불필요)**
- 튜닝 세션 ①로 구현이 시뮬레이션 수치를 재현하는지 확인 — 재현 안 되면 구현 버그다
- 오경보는 ②로만 잰다 (오답 접근이 없는 유일한 세션)

**최종 판정 — 홀드아웃 ③, 딱 한 번**
- 개선 **전 코드와 후 코드를 둘 다 한 번씩** 돌린다. ③은 촬영 후 한 번도 열지 않았다
- **합격선 = 개입 감지 90% 이상**(29회 중 27회) **이면서 오경보가 ②보다 악화되지 않을 것**

**회귀**
- `test_fsm`(11) · `test_hoi_sim`(4) · `test_imports`(2) 전부 통과 — `level` 기본값 2로 하위 호환
- **신규 단위 테스트**: 링 진입만으로는 차단되지 않는다 / 링에서 예열된 시간이 안쪽 진입 시 인정된다 / 갭메우기 0.3초 안엔 유지되고 이후 리셋된다
- ⭐ **실제 `run_demo.sh` GUI 기동 + 손 랜드마크·2단계 표시 육안 확인**

**중단 규칙**
- 홀드아웃에서 **90% 미달** → **달성했다고 쓰지 않는다.** 실측치를 그대로 기록
- 홀드아웃 **오경보가 ②의 2배를 넘으면** → 링 폭을 15px로 낮춰 재측정
- **ROI 오분류(엉뚱한 버튼)가 생기면** → 즉시 링 축소. **오분류는 미검출보다 위험하다**(FSM이 순서 위반을 통과시킨다, §9)

## 5. 범위에서 뺀 것 (YAGNI)

- **`FSM_DWELL_THRESHOLD_SEC`를 0.3으로 더 낮추기** — §6 참조. 별도 판단 사안이라 이 설계에 넣지 않는다.
- **링 폭을 박스 크기에 비례시키기** — 고정 25px는 거리에 따라 물리적 크기가 달라진다. 그러나 이번 촬영 거리에서 검증됐고, 비례식은 근거가 없다. 실콘솔·다른 거리에서 문제가 보이면 그때.
- **카메라 각도·전송 속도** — 별개 축(§10.21④).
- **오경보 억제 로직**(이동량 기반 "지나가는 중" 판정 등) — 현재 분당 1.1회이고 그마저 정상 동작에 가깝다. 필요해지면 그때.

## 6. 미결

- 🔴 **감지 ≠ 차단.** 선행시간 **≥0.5초는 14/26(54%)** 다. §9.4의 체류 임계 0.5초를 그대로 쓰면 **감지한 것의 절반은 차단이 늦는다.** 임계 하향(0.3초)은 오경보와 맞바꾸는 별도 결정이다.
- **선행시간 min 0.01~0.02초** — 어떤 임계로도 사전 차단이 불가능한 눌림이 매 세션 존재한다.
- **표본** — 눌림 26/23/29회는 여전히 작다. 90%와 80%를 통계적으로 구분하기 어렵다.
- **일반화** — 모조 콘솔·클린룸 아닌 실내·핫스팟 망·8~10fps 조건이다. 실콘솔·클린룸·저조도는 미검증.
- **링 폭의 거리 의존성** — §5 참조.
