# GPIO 눌림 로깅 — 설계 (2026-07-22)

> **구현 전 설계안.** 승인 후 구현한다(brainstorming-first HARD-GATE).
> 관련: [HOI 경로](2026-07-21-HOI-경로-design.md) §7 — 이 로깅이 그 도구에 **정답(ground truth)** 을 공급한다.

## 1. 왜 필요한가

버튼 누르는 동작을 촬영해도 **`bench_detector.py`에 GPIO 연동이 없어** "언제 눌렀는지"가 기록되지 않는다. 영상만 남는다.

### 없으면 못 하는 것

**① §9.4가 요구하는 값을 직접 못 잰다.** §9.4 재측정 계획의 문구가 *"사람이 버튼에 **처음 닿기까지**·버튼 간 이동에 걸리는 시간"* 인데, 이건 **눌림 시각이 있어야** 계산된다:

```
t_도착 = 손끝이 그 버튼 ROI에 처음 들어온 시각      (비전)
t_눌림 = GPIO 엣지 시각                            (물리)
선행시간 = t_눌림 − t_도착        ← §9.4가 원하는 값
```

없으면 영상을 눈으로 훑으며 "여기서 누른 것 같다"고 추정해야 한다. 오차가 크고 30초에 10회면 수작업이다.

**② ⭐ 사전 감지가 성립하는지 증명 못 한다.** 프로젝트 핵심이 **"버튼 누르기 직전에 사전 감지·차단"** 이므로 반드시 성립해야 하는 명제가 있다:

> **t_도착 < t_눌림** — 비전이 눌림보다 **먼저** 손을 인식해야 한다

GPIO 없이는 증명할 방법이 없다. 있으면 **"평균 X초 먼저 감지했다"** 가 수치로 나오며, 이는 **발표에 쓸 수 있는 핵심 근거**다.

**③ 오경보율을 못 잰다.** 누르지 않았는데 ROI에 머문 구간 = 오탐 후보. 안전 기능에서 이것이 오경보율이고, 눌림 기록이 있어야 "머물렀는데 안 눌렀다"를 구분할 수 있다.

## 2. 🔴 선행 확인 사항 (구현 전 반드시)

**모조 콘솔의 버튼이 실제로 GPIO에 배선돼 있는가?**

§12에서 전장 실물 결선·E2E 검증이 완료됐고(2026-07-15, 버튼 GPIO·릴레이·타워램프 전 구간 통과) `config.GPIO_BUTTON_PINS = {B1:5, B2:6, B3:13, B4:19}`·`GPIO_EMO_PIN=26`이 정의돼 있다. 그러나 **그 배선된 버튼과 지금 촬영 대상인 모조 콘솔(스티커 부착 박스)이 같은 물건인지**는 확인되지 않았다.

- **같다면** → 이 설계 그대로 진행
- **다르다면** → 촬영 세팅에서 **카메라에 보이는 버튼 = GPIO에 물린 버튼**이 되도록 구성해야 한다. 아니면 두 신호가 서로 다른 물건을 가리켜 측정이 무의미해진다

**이것이 가장 큰 위험이며, 구현보다 먼저 확인할 것.**

## 3. 변경 — 기존 파일 2개, 신규 0개

### 3.1 `Rpi5/Demo/test/bench_detector.py` (약 25줄 추가)

- **`--gpio` 플래그 추가** (기본 off — 기존 동작 완전 보존)
- 산출물에 **`<tag>_gpio_log.csv`** 추가: `timestamp, button, frame_no`
  - `timestamp`는 기존 로그와 **동일 형식·동일 시계**(`datetime.now()`) → 프레임과 바로 대조된다
  - `frame_no`는 눌림 시점의 현재 프레임 번호 → 영상 위치를 바로 찾을 수 있다
- `gpio_input.GpioInputController(on_button=…)` 를 그대로 재사용. **새 GPIO 코드를 쓰지 않는다**
- `manifest.json`에 `gpio: true/false` 기록

⚠️ **스레드 안전성** — gpiozero 콜백은 **내부 스레드**에서 불린다(`gpio_input.py` docstring 명시). 메인 루프가 `frame_no`를 증가시키는 동안 콜백이 읽으므로 **`threading.Lock`으로 보호**하고, 그 락 안에서 `csv.writerow`까지 수행한다(두 스레드가 같은 writer를 만지면 파일이 깨진다).

**fallback 보장**: `GpioInputController`는 gpiozero 미설치·핀 초기화 실패·비-Pi 환경에서 **예외 없이 로그만 남기도록 이미 설계돼 있다**(모듈 docstring). 따라서 `--gpio`를 켜도 GPIO가 없으면 촬영은 그대로 진행된다.

### 3.2 `Rpi5/Demo/test/dwell_probe.py` (약 40줄 추가)

- **`--gpio-log` 옵션** (기본: 세션명으로 자동 탐색, 없으면 기존 동작)
- 눌림 기록이 있으면 아래를 **추가로** 산출:

| 지표 | 계산 | 의미 |
|---|---|---|
| **선행시간** | `t_눌림 − t_도착` (눌림 직전 연속 ROI 구간의 시작) | §9.4가 원하는 값. 분포(median·min)로 임계 결정 |
| **사전 감지율** | `t_도착 < t_눌림` 인 눌림의 비율 | ⭐ 프로젝트 명제의 성립 여부 |
| **ROI 일치율** | 눌린 버튼 == 비전이 본 ROI 인 비율 | 엉뚱한 버튼을 가리키면 FSM이 오판한다 |
| **오탐 후보** | 눌림 없이 dwell 임계를 넘긴 구간 수 | 오경보율 |

- 기존 출력(ROI 시계열·공백 분포·갭메우기 전후 체류)은 **그대로 유지**

## 4. 거부한 대안

| 대안 | 거부 사유 |
|---|---|
| **영상에 눌림을 오버레이해 눈으로 판독** | 30초·10회면 수작업이 가능하나 **오차가 크고 재현이 안 된다.** 임계값은 문서 정본이 될 수치라 근거가 재현 가능해야 한다 |
| **`safety_console.py`(데모 GUI)로 촬영** | GPIO·비전·FSM이 이미 한자리에 있으나, **데모는 raw 프레임을 무손실로 저장하지 않는다**(녹화는 MJPG 손실압축). 재분석이 불가능하고, 시연용 코드에 계측을 넣는 것도 부담 |
| **별도 GPIO 로거를 따로 실행** | 프로세스가 둘이면 **시계가 어긋나고** 프레임 번호를 못 붙인다. 선행시간 계산의 정밀도가 그대로 손실 |
| **FSM까지 태워 실제 판정을 기록** | 최종 형태이긴 하나 지금은 **임계값을 정하려는 단계**다. 임계가 없는 상태에서 FSM을 돌리면 순환이다. 측정 → 임계 확정 → FSM 검증 순서가 맞다 |

## 5. 위험과 완화

| 위험 | 완화 |
|---|---|
| 🔴 **카메라에 보이는 버튼 ≠ GPIO 버튼** | **§2 선행 확인**. 세팅 단계에서 눌러 보며 로그가 찍히는지 확인 후 본 촬영 |
| 기존 촬영 동작 회귀 | `--gpio` 기본 off. 끄면 코드 경로가 종전과 동일 |
| 콜백 스레드 경합으로 CSV 손상 | 락으로 `writerow`까지 보호 |
| 디바운스로 눌림 시각이 밀림 | `GPIO_BOUNCE_SEC=0.05` → 최대 50ms 지연. 선행시간이 0.5초 규모라 **10% 이내**, 허용 가능. 결과 해석 시 명시 |
| EMO는 NC라 `when_released`에서 발사 | `GpioInputController`가 이미 처리(`gpio_input.py` docstring). 기록만 하므로 추가 처리 불필요 |

## 6. 검증

1. **회귀** — `--gpio` 없이 `bench_detector.py --frames 30` 실행해 산출물·검출 결과가 종전과 동일
2. **fallback** — `GPIO_INPUT_ENABLED=False` 또는 gpiozero 부재 상태에서 `--gpio`를 켜도 **예외 없이 촬영 완료**
3. **기록 정확성** — 촬영 중 버튼을 **의도적으로 3회** 누르고, `gpio_log.csv`에 3행이 찍히며 `frame_no`가 해당 시점 프레임과 맞는지(영상에서 육안 확인)
4. **시계 동기** — `gpio_log`의 timestamp가 `rawdet_log`의 같은 프레임 timestamp와 **±1프레임 이내**
5. **dwell_probe 통합** — 선행시간이 **양수**로 나오는지(음수면 t_도착/t_눌림 정의가 뒤집힌 것)
