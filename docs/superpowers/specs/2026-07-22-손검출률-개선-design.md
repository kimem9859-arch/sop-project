# 손 검출률 개선 — 사전 감지율 56% → 90% (설계)

- 일자: 2026-07-22
- 대상 repo: **Rpi5** (`Demo/hand_tracker.py`·`camera_thread.py`·`fsm.py`·`config.py`)
- 선행 문서: `2026-07-21-HOI-경로-design.md`(HOI 경로 확립) · `2026-07-22-detector-vdevice-공유-design.md`
- 임계 정본: 통합문서 **§9.4** / 카메라 판단: **§10.11**
- 🔴 **이 문서의 수치는 설계 근거용 진단값이다.** 확정 측정치는 사용자 확인 후 통합문서 §10에만 기록한다(단일정본 규칙).

## 1. 배경 — 왜 지금 이것인가

2026-07-22 카메라 연결 후 처음으로 **정답(GPIO 눌림)에 붙은 측정**을 했다. 버튼 누름 시뮬레이션 400프레임·눌림 16회를 촬영하고 `dwell_probe`로 대조한 결과:

**사전 감지율 9/16 (56%)** — 물리 버튼이 눌린 16번 중 7번을 비전이 **누르기 전에 보지 못했다.**

이 수치가 프로젝트의 핵심 명제("버튼 누르기 직전에 사전 감지·차단")를 직접 재는 값이다. FPS·검출률 같은 다른 지표가 아무리 좋아도 이것이 낮으면 시스템은 제 일을 못 한다.

같은 촬영에서 확인된 부수 사실:
- 버튼 검출(console_v2)은 건강하다 — B4가 296/400 프레임·median conf 0.894·B4↔EMO 오인 0회
- 놓친 눌림의 패턴이 뚜렷하다 — **B3 4회 전부·B2 5회 중 3회 미검출**, B1·B4는 전부 성공
- 선행시간 median 0.46s이나 **min 0.02s** — 가장 촉박한 건은 거의 동시였다

### 실패 지점 진단

`hand_tracker.detect()`의 실패 지점은 둘이다: ①팜 검출 0개 ②랜드마크 score < `HAND_MIN_SCORE`. 400장을 계측한 결과:

| 구간 | 프레임 | 비율 |
|---|---|---|
| ✅ 최종 성공 | 203 | 50.7% |
| 🔴 랜드마크 score < 0.5 | 110 | **27.5%** |
| 🔴 팜 검출 0개 | 87 | 21.8% |

랜드마크 score는 median 0.693이나 **p25가 0.360**으로 하위 사분위가 임계 아래에 깔려 있다.

⚠️ **"50.7%"를 성능 지표로 쓰지 않는다** — 400장 전부에 손이 있는 게 아니다(손을 뻗었다 뺐다 했다). 분모가 부정확하므로 기록만 하고 목표로 삼지 않는다. **목표 지표는 사전 감지율 하나다.**

### 파라미터만으로는 목표에 못 미친다 (실측)

손 추론 결과를 캐시해 조합별로 재평가했다(재추론 없음):

| 설정 | 사전 감지 | 오경보 후보 |
|---|---|---|
| **현재** (conf 0.5, margin 0, gap 0) | 9/16 (56%) | 1개 |
| conf 0.3 | 10/16 (62%) | 1개 |
| conf 0.3 + margin 25 + gap 0.5 | **14/16 (88%)** | 2개 |

두 가지가 결정적이다.

1. **갭메우기 단독은 효과가 0이다.** conf 0.5·margin 0에서 gap을 0.3이든 0.5든 넣어도 9/16 그대로다. 놓친 눌림 구간엔 **앞뒤로 이어붙일 검출이 아예 없어** 메울 갭이 존재하지 않는다. (설계 초기에 "갭메우기부터"를 추천했으나 이 측정이 반박했다 — §9.4 근거와 체류 구간 8→10개 개선을 보고 사전 감지율에도 통하리라 넘겨짚은 것이었다.)
2. **파라미터를 최대로 풀어도 88%**이고, 남은 2회는 **어떤 설정에서도 보이지 않는다**(손이 아예 검출되지 않은 프레임). 목표 90%는 파라미터 완화로 도달 불가다.

⭐ 다만 **margin 25에서도 ROI 오분류가 0회**였다(표의 "사전 감지"와 "ROI 일치"가 모든 조합에서 동일). 박스를 넓혀도 엉뚱한 버튼으로 오인하지 않았다 — margin 도입의 근거이나, **한 세션의 결과**이므로 홀드아웃에서 재확인한다.

## 2. 목표와 성공 기준

- **목표 지표 = 사전 감지율**(GPIO 눌림 중 비전이 먼저 본 비율). 정답에 붙어 있어 조작이 안 되고 핵심 명제를 그대로 잰다.
- **목표치 90%** — 홀드아웃 세션(눌림 24회) 기준 **22회 이상**.
- 부수 기록(목표 아님): 손 검출률 · 오경보 후보 수 · 선행시간 · HOI 포함 FPS.

## 3. 접근 — 인식 회수(A)가 본체, 파라미터(B)는 그 위에서

### 3-1. A: 프레임 간 트래킹 · ROI 유도 (`hand_tracker.py`)

현재는 매 프레임 **팜 검출 → 랜드마크**를 독립 실행한다. 원본 MediaPipe는 이전 프레임 랜드마크로 다음 프레임의 손 영역을 유도해 팜 검출을 매번 돌리지 않는다. `blaze_app_python`엔 그 되먹임이 **빠져 있다** — 모델 한계가 아니라 누락된 기능이다.

```
detect(frame):
  ① 직전 랜드마크 상태가 있으면
       landmarks_to_roi(직전 21점) → extract_roi → 랜드마크 추론    ← 팜 검출 생략
       score ≥ 유지임계 → 성공. 상태 갱신 후 반환
  ② ①이 없거나 실패하면
       팜 검출 → 기존 경로(현재 동작 그대로) → 성공 시 상태 저장
  ③ 연속 실패가 HAND_MAX_MISS 초과 → 상태 폐기 (드리프트 차단)
```

**`landmarks_to_roi(lms)`** 는 blaze가 이미 쓰는 규약을 그대로 재현한다. `BlazeDetector.extract_roi(frame, xc, yc, theta, scale)`는 팜 검출과 **독립적으로 호출 가능**하므로, 입력 네 값만 랜드마크에서 만들면 뒤쪽 파이프라인은 손댈 필요가 없다.

- 가상 팜 박스 = 손목(0)과 MCP들(1·5·9·13·17)의 바운딩 박스
- `theta` = 손목→중지MCP 벡터 각 − `theta0`(=π/2)
- `dscale`(=2.6)·`dy`(=−0.5) 동일 적용
- 상수 출처 = `blaze_common/blazeconfig.py`의 `palm_detect_v0_10_model_config`. 🔴 **복제하지 말고 로드한 모델 설정에서 읽는다**(값이 바뀌면 같이 따라가야 한다)

**기대 효과**
- 팜 검출 0개 **21.8%** → 팜을 거치지 않으므로 직접 회수 대상
- 랜드마크 저score **27.5%** → 크롭이 손에 더 맞으면 score 상승(간접·미확정)
- 속도: 팜 검출(5.7ms)을 대부분 생략 → **오히려 빨라진다**

**위험과 대비**

| 위험 | 대비 |
|---|---|
| 드리프트 — 손이 없는데 계속 따라감 | ③ miss 카운터 + 유지임계로 끊는다. 폴백(팜) 경로는 대조군 오검출 0.3%로 보수적임이 확인돼 있다 |
| 시연 경로 파손 | "모델 없으면 조용히 비활성" 설계 유지 + `HAND_TRACK_ENABLED=False` 한 줄로 현재 동작 즉시 원복 |
| 헛수고 | ⭐ **런타임 반영 전에 오프라인 선검증** — 캐시된 raw 400장에 새 로직을 적용해 사전 감지율이 실제로 오르는지 먼저 본다. 안 오르면 **여기서 멈춘다** |

### 3-2. B: 판정 경로 (A 완료 후 값 결정)

🔴 **B의 값은 A 이후에 재스윕해서 정한다.** 위 표의 최적값(conf 0.3·margin 25·gap 0.5)은 **A 이전의 최적**이라, A로 검출이 회수되면 달라진다. 지금 고정하면 필요 이상으로 느슨한 값을 굳히게 된다.

1. **`HAND_MIN_SCORE`** (`config.py`, 현재 0.5) — 재스윕 후 결정. 값만 바뀐다.
2. **ROI 마진** — `camera_thread.roi_at_point()`에 `margin` 도입(config 값). 지금은 손끝이 박스 **안에 완전히** 들어와야 판정되는데, 누르는 순간 손끝이 버튼에 가려지거나 살짝 어긋난다. ⚠️ `dwell_probe.roi_of`도 같은 규칙을 유지해야 한다 — 그 파일 주석이 "런타임과 동일해야 한다"고 못 박고 있다.
3. **갭메우기 0.3초** — §9.4 정본인데 **미구현**이다. 손이 잠깐 안 잡혀도 직전 ROI를 0.3초까지 유지. 같이 **`FSM_DWELL_THRESHOLD_SEC` 1.0 → §9.4의 0.5**로 맞춰 **코드-정본 불일치를 이 작업 안에서 해소**한다.
   - 구현 위치 = **`fsm.py`**. 갭메우기는 "끊긴 관측을 어떻게 읽을 것인가"라는 판정 규칙이고 §9.4가 dwell과 한 쌍으로 정의했다. `camera_thread`에 두면 ESP32·USB 두 클래스에 중복된다.

- 캐시 하네스로 A 적용 전/후의 사전 감지율 비교 → **안 오르면 런타임 반영 없이 중단**
- A 이후 파라미터 재스윕 → **오경보가 늘지 않는 선에서** 보수적 값 선택

**최종 판정 — 홀드아웃, 딱 한 번**
- 홀드아웃 = `20260722_154542_esp32_press-sim-holdout_console_v2` (400프레임·눌림 24회). 촬영 직후 **한 번도 열지 않았다.**
- 개선 **전 코드와 후 코드를 둘 다 한 번씩** 돌린다 → 기준선과 결과가 같은 자리에서, 훔쳐본 것 없이 나온다.
- **합격선 = 22/24 (90%)**

**회귀**
- `test_fsm`(11) · `test_hoi_sim`(4) · `test_imports`(2) 전부 통과
- 갭메우기·dwell 변경에 대한 FSM 단위 테스트 신규 추가(§9.4 값이 실제로 그렇게 동작하는지)
- ⭐ **실제 `run_demo.sh` GUI 기동 + 손 랜드마크 육안 확인.** 2026-07-22에 이것을 건너뛴 탓에 시연 경로가 `ImportError`로 깨진 채 커밋돼 있었다(`5f57f86`에서 수정).

**중단 규칙 (미리 정한다)**
- 오프라인에서 사전 감지율이 **안 오르면** → A 폐기, 원인 재분석
- 홀드아웃에서 **ROI 오분류가 생기면** → margin 되돌림. **오분류는 미검출보다 위험하다** — FSM이 순서 위반을 통과시킨다(§9)
- 홀드아웃 90% 미달이면 → **달성했다고 쓰지 않는다.** 실제 수치를 그대로 기록한다

## 5. 범위에서 뺀 것 (YAGNI)

- **카메라 전송 속도 개선** — 실측상 병목은 맞다(추론 32ms인데 실효 10.1fps, `cam_probe` 원시 전송도 8~11fps). 그러나 사전 감지율과는 별개 축이라 NFR-1 판정과 함께 따로 다룬다.
- **손 검출률을 목표로 삼기** — 분모(손이 있는 프레임 수)가 부정확하다. 기록만 한다.
- **트래킹 스무딩·칼만 필터** — 지금 문제는 "안 잡히는 것"이지 "떨리는 것"이 아니다.

## 6. 미결

- A의 유지임계·`HAND_MAX_MISS`·`landmarks_to_roi`의 크기 계수 — **오프라인 하네스로 교정**해서 정한다(설계 시점에 못 박지 않는다).
- margin의 일반화 — 오분류 0회는 **한 세션** 결과다. 홀드아웃이 1차 반증 기회이고, 실콘솔·클린룸에서 재확인이 남는다.
- 선행시간 min 0.02s 건 — dwell 임계를 아무리 낮춰도 이 눌림은 사전 차단이 불가능할 수 있다. 개선 후 이 값이 어떻게 변하는지 관찰 대상.
