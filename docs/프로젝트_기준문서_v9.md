- 기존 장비는 물리 버튼이 눌린 **'사후'에만 반응** → 이미 오작동한 뒤에야 인지하는 구조적 한계

**핵심 접근 (v3 — 핵심 시나리오 1개로 집중)**
- **단일 공정 시퀀스의 순서 위반 감지**: 정해진 공정 1종의 단계별 정답/오답 ROI를 기준으로, 잘못된 순서 시도를 버튼 입력 전에 잡아냄
- **FSM(유한 상태 머신) 기반 능동형 안전(에러 예방) 레이어**: 핵심 6단계 — IDLE(대기) → READY(준비) → PROCESS RUN(작업 진행) → MONITOR(감시) → WARNING(경고) → BLOCK(차단). 상세 전이는 §6.1 및 통합문서 §9 참조. ※ '능동형 Fail-Safe' 표현은 과장이라 지양(프레이밍 B, §1)
- **비전 구성 (핵심 = FPV 단일 비전)**: 핵심 범위는 FPV 웨어러블(1인칭) **단일 비전**으로 한정함. 고정형 Wide-View(3인칭) 주변 안전 감시를 더한 **하이브리드 구성은 §10 향후 확장**으로 분리 → 즉 핵심 시연 단계에서는 하이브리드가 아니라 단일 비전임(만능처럼 보이지 않도록 범위 명확화)
- **다층 피드백**: 시각(팝업) + 청각(부저/타워램프). ※ 촉각(진동 팔찌 햅틱)은 구현 범위 과다로 §10 향후 확장으로 이동(v5). 핵심 시연은 시청각 2층 피드백 + 인터록 차단으로 구성
- **인터록(Interlock)**: 오조작 강행 시 장비로 가는 전기 신호를 차단(인터록). ※시연은 실장비 없이 타워램프로 시각화. '원천 차단' 표현은 지양(프레이밍 B)

> **v3 차별점 (대결 아닌 플러스알파)**: 기존 콘솔은 물리 버튼이 눌린 *사후*에만 반응한다. 본 시스템은 그 위에 **버튼 입력 직전 사전 감지·차단 레이어를 얹는다.** 기존 장비를 대체·부정하는 것이 아니라, 기존 콘솔 위에 안전 레이어를 추가하는 구조다.

## 4. 설계 결정 및 가정 (Design Decisions & Assumptions)

> 이 섹션은 **"왜 이렇게 만들었는가"** 를 기록합니다. 향후 "왜 X 방식이냐"는 질문에 일관되게 답하기 위한 근거 모음입니다.

### 4.1 환경 가정 (Environment Assumptions)
- **작업 환경**: 반도체 클린룸
- **대상 설비**: 반도체 박막 증착 공정 중 **PECVD(플라스마 화학기상증착) 콘솔 장비** (가정). PECVD는 **CVD 계열의 한 변형**(PECVD ⊂ CVD, PVD와는 다른 계열)이며, §4.5의 4단계 시퀀스가 PECVD **정비(PM·웻클린 make-safe) 절차**에 대응함(2026-06-07 기동→정비 갱신).
- **스코프 한정 (v3 강화)**: 단일 장비 / 고정 화면 레이아웃 **1종** / **정해진 공정 시퀀스 1종의 순서 위반 감지**를 대상으로 함. 모든 장비·모든 공정 범용 동작은 목표가 아니며, 이는 PoC(개념증명)의 정상적 범위 설정임. → 물리 버튼 배치, 화면 그리드 위치 등 모든 좌표 가정이 이 단일 대상 전제 위에서 성립함.
- ✅ **시연용 "공정 시퀀스 1종" 단계 목록 확정 (v5, 2026-06-04)** — 4단계 공정으로 정의 완료. 상세는 **§4.5 시연용 공정 시퀀스 (정본)** 참조. (이전까지 [확인 필요]였던 항목 해소)

### 4.2 입력 방식 결정 (Input Method)
- **채택**: 아케이드 물리 버튼(다수, 공정 진행용) + 비상정지(EMO) 버튼 추가
- **상태**: ✅ **배치 확정(2026-06-09)** — 아케이드 버튼 **8개**(4색 흰·검·핑크·노랑 ×2), 좌 2×2 = 사용(B1~B4) / 우 2×2 = 미사용(확장), EMO 별도. 색↔단계 B1 노랑·B2 흰·B3 핑크·B4 검정. 상세 배치도는 통합문서 §5.3.
- **판단 근거**: 손-객체(버튼) 상호작용은 *"버튼이 눌렸다"* 는 명확한 디지털 이벤트로 판정 가능 → 깊이(Z)·선택 판정의 모호함이 없음
- **디스플레이의 역할**: 터치 모니터는 입력이 아니라 **출력**(공정 매뉴얼·레시피 안내, 경고 팝업 UI) 용도. 단, 경고/차단 **해제 버튼**은 디스플레이에 둠(§6.1 FSM 해제 경로 참조)

### 4.3 디스플레이 입력 방식 — 검토했으나 미채택 (요약)

> 마우스 포인터(A)·OCR(B)·위치 기반(C) 3방식을 검토했으나 **물리 버튼 채택** — 손-버튼은 "눌렸다"는 명확한 디지털 이벤트라 깊이(Z)·선택 판정 모호함을 회피하는 가장 확실한 방식. A·B는 해상도·방향성 한계로 미채택, **위치 기반(C)은 기각이 아니라 §10 향후 확장 후보**로 보존(물리버튼 대체는 깊이·선택 모호로 부적합).
>
> ※ 물리 버튼은 현장 재현이 아니라 **시연·구현용 의도적 스코프 단순화**다(실제 PECVD 콘솔의 조작 입력은 SEMI E95 기준 GUI가 주류). 3방식 상세 비교·근거는 통합문서 §13.2(마커 기반 화면 위치 인식).

### 4.4 핵심 판단 요약
- 채택 = **손-물리버튼**(깊이·선택 판정 모호 회피). 디스플레이 위치 기반(C)은 기각이 아니라 **물리버튼 병행 가능한 향후 확장 후보**(§10).

### 4.5 ✅ 시연용 공정 시퀀스 (정본 — 2026-06-04 확정 → 2026-06-07 정비 시퀀스로 갱신)

**4단계 + 비상정지**(PECVD 정비 PM·웻클린 make-safe): ① 클린·가스차단(플라즈마클린 CF₄/O₂ + SiH₄ 차단, **B1**) → ② 펌프/퍼지(N₂ ≥2사이클, **B2**) → ③ 전극 냉각(~300℃, **B3**) → ④ 챔버 벤트→개방(**B4**) + **EMO**(비상정지). 색↔단계: B1 노랑·B2 흰·B3 핑크·B4 검정.

- **판정 규칙(한 줄)**: 기대단계 N에서 **BN=정답 / 그 외 공정 버튼=오답.** 이 규칙 하나로 누락(뒤 버튼)·역순(앞 버튼)이 모두 잡힌다.
- **핵심 불변식**: 오답은 단계를 **진전시키지 않음.** 위반 BLOCK 해제 시 기대단계 **유지** / **EMO 해제는 기대단계=1로 리셋**(시퀀스 전체 중단·재시작).
- **⚠️ 콘솔 주석**: 콘솔엔 **디스플레이·물리버튼·타워램프·EMO만** 있고 **실제 PECVD 공정 기능은 없다**(시연용 모조품) — 단계명은 버튼 **라벨일 뿐.** 시연이 증명하는 것은 오직 "정해진 순서를 지켰는가/위반했는가"의 비전 감지·차단(§4.1 범위 한정 전제와 정합).

> 📍 **단계별 정답/오답 표·전이 상세·사용자 흐름·임계값 정본(타이머 dwell 0.5초+갭 0.3초, 응답 ≤100ms, HOI 0.70/0.55, 작업자 본인 해제)은 통합문서 §6·§9.4.**

#### ✅ FSM 정본 (6단계 — 2026.06.03 확정)

**핵심 6상태**: **IDLE**(대기) → **READY**(준비) → **PROCESS RUN**(작업 진행) → **MONITOR**(감시 — 정답/오답 분기) → **WARNING**(시·청 경고) → **BLOCK**(인터록 차단).

- **요지**: MONITOR에서 오답 ROI 체류 시 **타이머로 스침을 거른 뒤** WARNING, **오답 강행/EMO는 즉시 BLOCK**(경고 생략). 해제 버튼 **2종**(WARNING→MONITOR / BLOCK→READY, 둘 다 디스플레이).
- **EMO 구분**: 위반 BLOCK 해제는 기대단계 **유지**, **EMO 해제는 기대단계=1로 리셋**(§4.5).
- 과거 5·7·9단계 혼재는 **폐기**(§11).

> 📍 **상태 전이도·전이 시나리오·필터링/타이머 상세는 통합문서 §9.** FSM 정본 = 통합문서 §9의 6상태 정의 + 본문 Mermaid 상태도(별도 `.drawio` 파일은 팀 공유 드라이브 보관, 정본 아님).

## 8. 프로젝트 진척 · 관리 (요약 · 이력)

> 진척·작업 상태는 이 문서의 정본이 아니다(일정·작업 추적 도구 미사용). 아래는 재빌딩 배경의 **요약·이력**만 남긴다.

### 진척 재평가 (요약)
- 상반기(2~5월)는 지연·이월·준완성이 많았고(완료 확정 2건: `person_v1` 재학습 mAP50≈0.96 · YOLO 클린룸 인식 테스트), **6/3~ 프로젝트 리빌딩**(방향성·기준문서·통합문서 재정비)으로 재출발했다.
- 지연 배경(2차 물품 6주 배송 지연·1차 부품 성능 부족·기획 완성도·학사 일정)은 **통합문서 §2.6 회고** 참조.
- 개별 작업의 진행/지연 상태는 이 문서에서 다루지 않는다(도구 미사용 — 나침반 유지).

### 프로젝트 관리 체계 (애자일)

> ⚠️ **일정·작업 추적 도구는 사용 중단**(Linear·Google Calendar). 아래는 재계획 당시(2026-06-04) 채택했던 관리 방식의 **기록**이며, 현재 일정·작업 상태는 이 문서에서 다루지 않는다(문서 범위 밖).

- **방식(당시)**: **2주 사이클(스프린트)** · Product Backlog 우선 · Sprint 1부터 정교화하며 반복
- **Product Backlog 52개(당시)**: 핵심 28 + 향후확장 9(§10) + 기획·분석 7(과거 완료/이월/리빌딩) + 최종모델 학습 1 + 한이음 5 + 1학기 발표 2
- **추정**: Fibonacci(1·2·3·5·8) Story Point / **우선순위**: Urgent · High · Medium · Low
- 개별 작업의 진행/지연 상태는 이 문서에서 다루지 않는다(나침반 유지). 본문 곳곳의 `PRO-##` 표기는 당시 백로그 번호의 **이력 흔적**이다.

### 해결 완료한 주요 기술 문제 (이력 — 유효)
- YOLO 모델 → Hailo HEF 변환 후 추론 시 객체 미탐지/오라벨 현상
- 원인: NMS 방식 불일치 / 입력 형식 불일치(FLOAT32 vs UINT8) / 클래스 수 불일치 / 파이프라인 호환성
- 해결: Python 직접 처리 전환, 입력 형식 UINT8 통일, 클래스 14→10개 이하 재맵핑, 검증된 YOLO 기본 모델 적용
- ※ 이는 초기 `person_v1` 계열 변환 경험이며, `console_v1.hef`는 빌드·파이 실추론 완료(B4 미탐지 → console_v2 재학습 확정), 최종 `console_v2.hef` 변환은 추후.

## 10. 향후 확장 (Future Extensions)

> v3 범위 축소로 핵심에서 분리된 기능들을 **폐기가 아니라 여기 보존**합니다. 학기 내 핵심 시나리오 완수 후, 또는 후속 연구에서 확장 가능한 항목입니다. (과거 백로그 `향후확장` 라벨 — PRO-29~37, 도구 미사용)

### 10.1 v3에서 핵심 → 확장으로 이동한 항목
- **주변 안전 감시 (Wide-View 독립 스레드)**: 고정형 웹캠으로 작업자 안전 구역 이탈·충돌 위험 객체를 FSM 공정 루프와 독립적으로 병렬 감시. (4월 초안의 "System B / Phase α")
- **로깅 (오류 데이터 기록·활용)**: 작업 오류를 '단순 망설임(숙련도 부족)'과 '절차 위반'으로 분류, 발생 시간·ID 등과 함께 CSV/시스템 로그로 기록 → 맞춤형 재교육 자료로 활용. 관련 KPI(3종 로그 분류, 10개 이상 데이터 항목)도 이 영역. (※ FSM 순서도에서 '정상 로그 저장' 노드 제거됨)
- **작업 완료 확인 제스처**: MediaPipe Gesture Recognizer의 `Thumb_Up` 인식으로 작업 완료 시 화면/프레임 캡처 트리거. 지적확인환호(指差喚呼) 원리 차용. ※ MediaPipe는 제스처 인식만, 캡처 저장은 앱 코드로 구현 필요.
- **FSM CAUTION (경계) 상태**: 임계시간 초과 시 타이머 리셋 / 경고 후 손 회수 시의 중간 상태. 핵심 6단계에서 제외, 확장 시 복원 가능.
- **FSM 관리자 승인 에스컬레이션 (①②③회 카운트)**: 오류 누적(1회→2회→3회) 시 관리자 승인을 거쳐 경고 해제·카운트 리셋하는 상위 안전 로직.
- **기타 휴먼에러 시나리오**: 화학물질 취급, 위험구역 침입, 보호구 착용 확인, 공정 식별(FOUP 8"/12"), 금지 물체 접촉 등. (핵심 시나리오 '순서 위반' 외 후보들)
- **촉각 피드백 (진동 팔찌 햅틱)** *(v5 이동)*: XIAO ESP32-C3 + 코인 진동모터로 위반 감지 시 BLE 햅틱 알림. 다층 피드백의 세 번째 층(시각·청각에 더하는 촉각). 구현 범위 과다(BLE 통신 안정화 + 펌웨어 + 3D 팔찌 제작)로 핵심 시연에서 제외, 확장 시 복원. 관련 KPI: 햅틱 응답 ≤ 200ms. 부품은 이미 보유(§5).

### 10.2 기존 확장 항목 (v2에서 유지)
- **마커 기반 화면 위치 인식 (디스플레이 입력 방식 경로)**: 모니터 모서리에 ArUco/AprilTag 마커를 부착 → 원근 보정(호모그래피)으로 화면을 정면 좌표계로 펴고 → 그리드 칸과 손 위치를 매핑. 글자 해상도 벽을 회피. 장비가 바뀌어도 알고리즘 재작성 없이 **"장비별 1회 캘리브레이션"** 으로 확장 가능. (§4.3 방식 C의 발전형)
  - **선결 과제**: 단안 카메라 깊이(Z) 판정, 허공의 손 '선택 이벤트' 판정 모호함 해결.

### 10.3 별도 산출물로 분리 (FSM 순서도에서 제외)
- **WiFi 접속 흐름**: 스마트글라스 전원 ON → WiFi 통신모듈 → 접속 성공/오류 → 1회 재접속 → 핫스팟 IP 고정 → 카메라 영상 전송. 이는 FSM 상태 전이가 아니라 **통신 초기화 시퀀스**이므로, FSM 순서도에서 분리해 **별도 시스템 다이어그램**으로 관리한다. [작성 예정]
